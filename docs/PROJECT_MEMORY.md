# Project Memory

## Stable Context

- Project goal: build an independent StabX-like coarse GPS-denied visual return system.
- Target hardware: Raspberry Pi Zero 2W class companion computer, Pi Camera or thermal camera, Matek H743 Slim V3 / ArduPilot flight controller.
- Repository strategy: keep the imported baseline under `reference/` and build the new flight-critical system under `core/`.
- Core implementation direction: C++ replay-first architecture with deterministic timing, bounded queues, and explicit health/failsafe state.
- Current validated core baseline: replay pipeline plus route signature v1, coarse matching, initial navigation command model, dry-run MAVLink boundary, and hardware capture boundary has `ReplayFrameSource`, `Gray8ResizePreprocessor`, `HealthMonitor`, `PipelineHarness`, binary `VHRS` route signature writer/reader, route inspection summary, stateless route self-match artifact check, stateless route perturbation artifact check, offline route distinctiveness diagnostics, `RouteSignatureRecorder`, `Gray8RouteMatcher`, `BoundedNavigator` with yaw-rate slew limiting, `DryRunCommandSink`, `DryRunMavlinkBridge`, `MavlinkTelemetryAdapter` with heartbeat freshness plus armed/Guided permission gates, `PiCameraSource` with initial libcamera backend behind compile-time `VISUAL_HOMING_ENABLE_LIBCAMERA` and runtime `enable_live_capture`, active camera profiles, live camera route recording/matching, read-only MAVLink telemetry capture/inspection/validation/streaming, live dry-run command quality metrics, live-output safety-gate diagnostics, default zero-forward-speed and zero-lateral-speed live-output gate policy, non-live live-output audit-log boundary, non-live live-output session scaffold with lazy bridge start and audit-before-send ordering, real-file session audit smoke coverage, opt-in Pi live-route session audit artifact support, route-quality/readiness log checker support, audit-log checker support, UTC wall-clock operator logging, MSVC/CMake build support, Raspberry Pi bootstrap/test scripts validated on Pi Zero 2W class hardware, and 23 passing CTest tests. The latest accepted `jtzero` endpoint-stop bench props-off run at `375f6cd` validated the default/pre-attach wrapper after the explicit serial-writer attach flag landed: Pi CTest passed 23/23, `live_output_writer_attached=false`, 117/225 live matches before endpoint stop, directional and endpoint progress gates, read-only telemetry health with zero dropped bytes, 117/117 valid dry-run commands, expected live-output blocking with `live_output_unavailable:117`, and a session audit artifact with `allowed=0 blocked=117 stop_reason=endpoint_progress_reached`.
- Near-term priority: keep live MAVLink output blocked while implementing only the reviewed plan for the first bench props-off live-output boundary. Phase 1 split the compile-time boundary: default builds keep output disabled, `VISUAL_HOMING_ENABLE_LIVE_MAVLINK_OUTPUT=ON` is invalid without `VISUAL_HOMING_ENABLE_BENCH_PROPS_OFF_LIVE_OUTPUT=ON`, and the paired bench-scope build keeps `VISUAL_HOMING_LIVE_MAVLINK_OUTPUT_AVAILABLE=0` unless the separate `VISUAL_HOMING_ATTACH_BENCH_PROPS_OFF_SERIAL_WRITER=ON` flag is also set. Phase 2 added an injectable `LiveMavlinkCommandWriter` interface behind `LiveMavlinkBridge` with fake-writer tests. The first concrete serial writer library boundary now exists: `LiveMavlinkSerialCommandWriter` emits MAVLink2 body-frame yaw-rate-only commands through an injectable byte transport, zero-fills but masks velocity fields, rejects invalid/non-finite/nonzero-velocity/out-of-bound commands before write, and can be wired into runtime live-route sessions only by the explicit attach flag. Phase 3 added runtime/operator/max-limit scaffolding: the live-route audit path can accept explicit runtime enable, exact props-off confirmation, max commands, and max seconds, and `LiveMavlinkOutputSession` stops before writer send when those limits are reached. Phase 4 added an explicit `live_output_available` safety gate wired to the compiled bridge availability state for runtime-controlled live output; default runtime builds still block with `live_output_unavailable`. Phase 5 extended the audit contract so command records include `decision=allowed|blocked`, telemetry context, route-match context, and checker-level expected allowed/blocked counts while preserving old blocked-readiness logs. Phase 6 added `scripts/run-live-output-bench-props-off-pi.sh` as the dedicated bench command; it requires the exact props-off confirmation and now uses endpoint-complete stop with variable-length `auto` checker expectations. Pre-attach hardening keeps blocked sessions from opening the writer in unavailable builds, enforces zero lateral speed, requires audit before send, stops explicitly on bridge/audit/send failures, and stops command generation at endpoint completion. Default desktop/Pi test scripts now explicitly force all live-output CMake flags `OFF` for ordinary builds so stale CMake cache state cannot silently attach the writer; reviewed Pi attach builds use explicit env flags and a separate build directory by default. Fail-closed wrapper evidence is accepted at commits `d355bf1`, `6fc9cd2`, `98d5407`, and `375f6cd`; the `375f6cd` run proves the ordinary wrapper still leaves `live_output_writer_attached=false` after the explicit attach flag landed. `docs/LIVE_OUTPUT_SAFETY_PLAN.md` is the controlling safety artifact and `docs/LIVE_OUTPUT_BENCH_PROPS_OFF_PLAN.md` defines the next implementation scope: bench-only, propellers removed, yaw-rate-only, zero forward/lateral velocity, explicit compile-time/runtime/operator gates, audit readiness, single-writer ownership, telemetry freshness, route-match freshness/confidence, command quality, and hard max command/duration limits. `docs/LIVE_OUTPUT_READINESS_RECORD.md` records 3/3 clean Pi dry-runs plus the accepted fail-closed bench props-off runtime evidence. Remaining non-live work includes timestamp-aligned continuous MAVLink telemetry integration, an HTTP wrapper over the profile registry, altitude-aware route metadata, thermal normalization/profile support, and then reviewed attach-build Pi validation without enabling flight. Milestone 7 flight ladder is deferred until the bench props-off live-output boundary is separately implemented and reviewed.
- StabX-like target remains coarse GPS-denied visual return, not metric SLAM or precision hover: long-distance behavior depends on camera/FOV profiles, attitude/altitude telemetry, altitude/scale-aware route metadata, thermal/visible camera profiles, route segmentation, reacquire behavior, and conservative confidence gates. Barometer altitude and visual image-scale changes are useful as dry-run scale diagnostics first; they are not command authority until compared against route evidence and covered by a separate safety decision. A later visual brake / station-keeping assist mode is allowed as a separate milestone only: start dry-run-only, use FC IMU/attitude to compensate image displacement, keep ArduPilot responsible for real hover/hold/failsafe, and require deadband/gain/rate/timeout gates plus a reviewed scale source before any live counter-command authority.
- Python/reference code is useful for documentation, tools, diagnostics, UI concepts, and comparison, but should not become the new flight-critical scheduler.
- Commit messages should be detailed and explain what changed, why, impact, validation, and risk.

## Working Rules

- Technical decisions go to `docs/DECISIONS.md`.
- Current progress goes to `docs/SESSION_LOG.md`.
- Stable project context goes to this file.
- Ideas and research notes go to `notes/`.
- Code changes are captured in detailed git commits.
- Generated build artifacts should stay out of git; prefer reproducible scripts/docs for local validation.
- Every completed task must be pushed to GitHub after the local commit and validation are complete.
