# Project Memory

## Stable Context

- Project goal: build an independent StabX-like coarse GPS-denied visual return system.
- Target hardware: Raspberry Pi Zero 2W class companion computer, Pi Camera or thermal camera, Matek H743 Slim V3 / ArduPilot flight controller.
- Canonical hardware/access source-of-truth: `docs/HARDWARE_ACCESS_BASELINE_UA.md`. Read it before Pi, SSH, Matek, serial, ArduPilot parameter, FC/JT_Zero acceptance, or external-nav hardware work. It separates confirmed facts, operator-reported recollection, currently-offline state, and values that require live verification.
- Canonical coordinate-frame contract: `docs/COORDINATE_FRAME_CONTRACT_UA.md`. ArduPilot local pose is NED, body/twist is FRD, ROS local pose is ENU, ROS body is FLU, and Visual route progress remains `ROUTE_FRD` until an explicit route origin/heading projection produces `LOCAL_NED`. The current `VISION_POSITION_ESTIMATE` path fails closed unless route alignment, altitude-origin alignment, and an independent yaw observation are explicit. FC `ATTITUDE.yaw` remains diagnostic only because the captured FC uses `EK3_SRC1_YAW=6` (`ExternalNav`). For the confirmed-straight forward route, yaw now uses `wrap(route_heading_ned_rad + direction_error_rad)`, where the residual is independently estimated by bounded horizontal image shift; disabled, non-finite, invalid-match, or boundary-saturated observations produce `yaw_source_not_independent`. Stored route heading hints are not authority. `VHRS v1` lacks metric `x/y` route geometry, so the straight-axis progress model remains artifact-specific; geographic route bearing, explicit WGS84 EKF origin, altitude-origin alignment, and reverse-yaw semantics remain blockers before acceptance. MAVLink `ODOMETRY` frame selection is not yet implemented.
- Repository strategy: keep the imported baseline under `reference/` and build the new flight-critical system under `core/`.
- Core implementation direction: C++ replay-first architecture with deterministic timing, bounded queues, and explicit health/failsafe state.

- Architecture principles: keep module boundaries explicit and replaceable across capture, preprocessing, route artifact I/O, matching, telemetry, navigation, command output, audit, and safety gates. The companion is operator-in-the-loop command assist during the current safety stages, not an autonomous vehicle controller; ArduPilot remains responsible for stabilization, failsafe behavior, mode management, and motor mixing. Route artifacts and MAVLink serial bytes are untrusted inputs until validated, freshness-checked, and gated. Future sensors/algorithms such as thermal cameras, rangefinders, optical flow, VIO/UWB, visual-scale diagnostics, and stronger matchers should plug in behind interfaces/config rather than rewriting the scheduler.

## Working Rules

- Technical decisions go to `docs/DECISIONS.md`.
- Current progress goes to `docs/SESSION_LOG.md`.
- Stable project context goes to this file.
- Ideas and research notes go to `docs/VISUAL_HOMING_IDEAS_UA.md` when they affect Visual Homing behavior, safety states, RTL/HOVER, endpoint handling, or operator UI. Short raw notes can still go to `notes/`.
- Curated extracts from pasted previous-session transcripts go to `docs/SESSION_EXTRACTS_BANK_UA.md`; do not store raw chat there, preserve only reusable decisions, artifacts, commands, pitfalls, and open questions.
- Hardware, wiring, SSH and installed-FC facts go to `docs/HARDWARE_ACCESS_BASELINE_UA.md`; never store private-key contents or passwords there, and never present historical evidence as current hardware state without a new verification.
- Before any Pi field/send/FC-JT_Zero probe run, run `./scripts/check-pi-field-readiness.sh` from the Pi checkout and proceed only on `visual_homing_pi_field_readiness_done passed=true`; fix named blockers instead of running Visual Homing wrappers with `sudo`.
- Code changes are captured in detailed git commits.
- Generated build artifacts should stay out of git; prefer reproducible scripts/docs for local validation.
- Every completed task must be pushed to GitHub after the local commit and validation are complete.
